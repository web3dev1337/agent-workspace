## TL;DR

GPT-5.6 Soul is a big quality jump but burns usage per message roughly 10x faster than GPT-5.5 (up to ~15% of a 5-hour limit in one message on xhigh/max, vs 0.1–2% typical on 5.5). Most of the burn is avoidable: skip Ultra, turn off fast mode, default to **high** (or medium) reasoning instead of xhigh/max, rein in sub-agent spawning via AGENTS.md, and control run length by putting explicit stop points in prompts. Don't hand-tune context/compaction settings — OpenAI tuned those correctly and overriding them makes the model dumber and costs more.

## Why 5.6 burns so much more than 5.5

- 5.5 constantly stopped mid-task to ask for permission/feedback, so each message used little quota (~0.1–2% of a 5-hour limit). Its weaknesses (context contamination, bad compaction) forced short threads.
- 5.6 fixed the premature stopping — it now runs autonomously for hours per prompt (Theo's team reports 8h+ single threads without `/goal`). That's why per-message usage exploded: same total work, but concentrated in far fewer, far longer messages.
- Theo hit his first limit 20 minutes after his pre-release free-usage period ended — triggered by the new **Ultra** feature.

## Rate-limit mechanics (as of filming)

- Subscriptions have a 5-hour rolling limit and a weekly limit (≈4–5x the 5-hour limit).
- OpenAI **temporarily removed the 5-hour limit** for Plus/Business/Pro, leaving only the weekly limit. Upside: fewer interruptions. Downside: nothing stops a single runaway (e.g. Ultra) prompt from eating a large chunk of your week.
- Theo's own trick for the 5-hour window: a cron job fires a tiny prompt every 5 hours so a window is always ticking. (Same idea as the quota-warmer already running on this machine.)

## Recommendations (in order of impact)

1. **Biggest one — define the stop point in the prompt.** 5.6 keeps going unless told where to end. Pattern: "build X, write a plan, then STOP and ask for feedback" — or place the stop much later ("implement, test with computer use, open a PR, address the first round of review comments, then stop"). Choosing the endpoint via prompting beats trying to control cost via reasoning levels or harness changes.
2. **Don't use Ultra.** Not worth it in its current state (dedicated follow-up video promised). It's a feature, not a reasoning level.
3. **Turn off fast mode.** 1.5x speed for 2.5x limit burn. Acceptable when a 5.5 message was 2% of a limit; brutal when a 5.6 message is 15% (≈ half a 5-hour limit for one message). With 5.6 running unattended anyway, inference speed barely matters — tool calls/tests dominate wall-clock.
4. **Reasoning level: default to high; medium is also excellent.** Benchmarks he cites (SWE-style bench, per-task API cost as proxy for quota burn):
   - Soul low: 45% @ ~$1.00 · medium: 61% @ $1.86 · high: 69% @ $3.47 · xhigh: 71% @ $4.70 · max: 73% @ $8.39
   - Claude Fable 5 on the same bench: 70% @ ~$13/task — so Soul-high matches Fable at ~1/4 the cost, and on the intelligence-vs-cost index Soul-high is the one point in the "good" quadrant.
   - Above high, cost roughly doubles for +2–4 points — not worth it. (Caveat: Cursor Bench showed a bigger high→max gap, and Fable 5 beat Soul there — workload-dependent.)
   - 5.6 self-regulates: on simple tasks, high reasoning uses barely more tokens than low, so leaving it on high is safe.
   - Anecdote for medium: the OpenCode team's reasoning-level config was silently broken (the API key for reasoning level changed and many tools don't pass it through), so their entire month of "favorite model ever" usage was actually **medium**.
5. **Sub-agents: use with caution.** 5.6 was trained to spawn them eagerly, and Codex's two implementations (V1/V2) are both weak right now. If you're burning limits from sub-agent fan-out, add to global AGENTS.md: *only use sub-agents if the user explicitly requests them*. Claude Code's and Cursor's sub-agent implementations are meaningfully better — Theo runs his Codex sub inside Claude Code ("Claudex", using Soul as the model), publicly blessed by Tibo.
6. **Model selection:** Soul for everything (high default, low/medium for simple work). Luna is for programmatic/API filtering tasks, not coding. Terra disappointing — artificial-analysis data shows Soul and Luna beat it at every intelligence-vs-cost point.
7. **Do NOT manually cap the context window / auto-compact threshold.** Circulating advice says to lower it to save money — it does the opposite: the model was trained around the default compaction level, compaction itself is expensive, and OpenAI doesn't charge extra above 270K context. Confirmed wrong by Tibo directly.

## OpenAI-side fixes announced during/around filming (Tibo)

- 5-hour limits temporarily removed; a usage reset shipped for everyone (~6M active users).
- Inference optimizations landing: ~10% cheaper.
- Bug found: raising product context 272K→372K charged more usage than intended — reverted to 272K until fixed; should noticeably slow usage drain.
- Confirmed the leaked "juice values" change was real but has been reverted.
- Fixing over-eager multi-agent use on high/xhigh, plus auto-review efficiency.
- Theo's take: it's a moving target, but every recommendation above still holds.

## Meta-advice

Don't install someone else's "oh-my-X" config bundle and hope. Experiment yourself: rewrite your own AGENTS.md / CLAUDE.md by hand, read agent traces when behavior surprises you, and adjust until it feels right — that's how you build intuition for where to put the stop signs.
